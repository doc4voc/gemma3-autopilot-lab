1. Executive Summary

最重要の観察
「LLM単体」よりも、技能(スキル)分解と外部メモリが性能を安定化
再現性の鍵
閉ループ: 観測→行動→結果→反省 をログで回す
安全性の鍵
「主役はLLM」でも 安全シールド は併用が一般的

提案の骨子（このプロジェクト向け）

  • スキルAPI化: LLMはマクロ行動を選び、低層は実行器が担う（SayCan / Voyagerの系譜）。
  • メモリ=地図ツール化: ヒートマップを「問合せ可能な外部知識」にしてLLMへ供給（Tag Map / VLMapsの系譜）。
  • 反省と再計画: ループ・壁接触・ターゲット接近失敗を「反省メモ」として次周期に反映（ReAct / Reflexion / Inner Monologue）。
  • 評価の標準化: 成功率＋経路効率（SPL系）＋安全違反率で比較可能にする（Embodied navigation評価）。

2. Context: Current System (as implemented)

本節はコードから確認した要点（主に src/services/explorationMemory.js, src/services/ollamaService.js）。

Component Current behavior Implication for related-work reuse
Observation to LLM 8本センサ距離、ターゲット距離/角度、ターゲットヒット、Pose(X,Z,Heading)、速度、スタック、履歴、
Exploration Memory(RAG)として frontier/risky/currentCell/sectorSafety/loopRate を圧縮して供給。 		LLMは「自位置(ix,iz)」と「向き(Heading)」を持つため、地図/候補を言語的に解釈可能。 ただしフルラスタではないため、局所パッチの表現が性能を左右。
Exploration Memory update 2mセル。全レイを統合し、レイ上のセルを openHits（自由空間の証拠）として加算。ヒット点セルへ obstacleHits と riskEMA を加算。ループ率(loopRate)も算出。 古典的な「占有格子/コストマップ」系の設計思想と整合。frontier定義を強化すると探索効率に直結。
Safety guard センサ遮断・メモリ遮断（sectorSafety）を用いて forward を抑制し、代替セクタへ偏向。近距離ターゲット接触だけは例外的に前進許容。 多くの実ロボ/Embodied系は、LLM主役でも安全シールドを併用する。実験上は「どれだけ介入したか」を指標化すると議論しやすい。

4. What We Can Reuse: Design Patterns

4.1 Recommended Control Stack (LLM as main, tools as constraints)

主張: 「LLMが主役」でも、スキル実行器・安全シールド・外部地図は “意思決定の代替” ではなく “意思決定を支える構造” として扱うと再現性が上がる。

4.2 Patterns distilled

Pattern What it buys Minimal adaptation here Linked works
Skill library (macro actions) 連続制御のブレを減らし、出力フォーマットも安定 LLM出力を skill + params に寄せ、実行器で throttle/steer に落とす SayCan [R1], Voyager [R2]
Closed-loop reflection memory 同じミス（壁へ向かう、ループ）を繰り返しにくくする 「結果要約（距離変化/衝突/ループ率）」を短文で次周期プロンプトに組み込み、反省欄を保存 ReAct [R3], Reflexion [R4], Inner Monologue [R5]
Map as queryable tool LLMが「どこを探索したか」を言語で扱える ヒートマップを “説明可能な断片” に圧縮（frontier/top-risk/local-patch）し、参照しやすくする Tag Map [R6], VLMaps [R7]
Standardized embodied metrics 改善が「主観」ではなく指標で議論できる Capture成功率 + 経路効率（SPL類似） + 安全違反率 + ループ率 Embodied eval [R10]

5. Recommended Improvements (Prioritized)

重要: 「プログラムで直接ターゲットを決める」方向は避け、LLMが判断するための 情報設計 と 行動インターフェース を整える。

Priority Proposal Why (failure mode it targets) Implementation sketch Evidence
P0 Action abstraction: “skill selection” output JSON/連続制御の揺れで壁へ突っ込む、あるいは小刻み前後で停滞しやすい。
まず行動空間を整理すると、LLMの判断が安定する。 		LLM出力に skill フィールドを追加（例: APPROACH_TARGET, SCAN_SECTOR, BACKOFF_AND_TURN, MOVE_TO_FRONTIER）。
実行器が duration/steer/throttle の波形を生成。LLMはスキルとパラメータに集中。 		SayCan [R1], Voyager [R2]
P0 Reflection slot + outcome summary 壁方向へ向かうのは「直前の意思決定が失敗した」ことをLLMが学べていない可能性。
1周期前の結果を短い要約で与えると再発が減りやすい。 		drive-logに「outcome」を追加: 目標距離の増減、最小障害物距離、loopRate変化、ガード介入有無。
次プロンプトへ 2-3行で注入し、reflection をJSONで返させて保存。 		ReAct [R3], Reflexion [R4], Inner Monologue [R5]
P1 Local map patch (5x5) with quantization 現状はTop-K候補中心で、周辺の「壁の面」や「袋小路」を表現しきれない可能性。
局所パッチを量子化して渡すと、方向判断の誤りが減る。 		currentCellを中心に 5x5 の risk/open/obstacle を 3段階程度に量子化（例: LOW/MID/HIGH）。
文字列行列として短く提示（トークン節約）。 		Map-as-tool系 [R6][R7]
P1 Frontier definition upgrade (classic) “重みが低い方へ” だけだと、境界外方向や未探索の袋小路に引き寄せられやすい。
Frontier（既知freeと未知の境界）を強化すると無駄探索が減る。 		openHits/obstacleHits を用いてセルを {FREE / OCCUPIED / UNKNOWN} に分類し、FREEに隣接するUNKNOWNをfrontier候補へ。
その候補リストを LLM へ渡し、選択はLLMに任せる。 		Frontier exploration [R11]
P2 Embodied metrics: Capture-SPL 「速度」ではなく「到達ルートの最適化」を見たい。
SPL系は “最短経路にどれだけ近いか” を測れるため、目的に合う。 		1エピソードで、成功時のみ SPL = shortestPath / pathLength を計算（shortestは近似でも可）。
Capture成功率・衝突率・loopRateと一緒にレポート化。 		Embodied eval [R10]

注: PaLM-E/RT-2は「視覚を含む」方向の将来拡張候補。現時点の8レイ＋数値系でも、上記P0/P1だけで改善余地が大きい。

6. Evaluation Plan (paper-grade)

研究として通すためには「改善がどの失敗モードに効いたか」を切り分ける必要がある。以下は最小のA/B計画。

Experiment Change Primary metrics Expected effect
AB-1 現状 vs P0 Skill selection Captures/min, collision rate, parse-error rate 壁突進・小刻み前後の減少、JSON安定
AB-2 P0 only vs P0+Reflection (P0) loopRate, same-cell revisit ratio, safety interventions ループ再発の減少、復帰の早期化
AB-3 P0+Reflection vs +Local patch (P1) wrong-way-to-wall events, no-go chosen rate 方向誤りの減少、探索の無駄減
AB-4 Frontier upgrade (P1) Coverage efficiency, Capture-SPL (success only) 境界外へ吸われる傾向の低減

ログ要件（最低限）

  • LLM: prompt / raw response / parse method / schema version
  • Decision: mode/sector/skill/params/confidence + outcome summary
  • Safety: guardApplied, guardReason, override ratio
  • Memory: currentCell(ix,iz), loopRate, sectorSafety, frontier top-k, (optional) local patch
